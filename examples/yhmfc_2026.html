1故事：为什么有这场比赛

气象预报是国家级能力，但模式预报和站点实况之间永远有系统性偏差。雅安身处川西山地，地形雨与夜雨主导，是预报最难的地区之一。

类比：买菜 vs 真实价格

NWP（数值天气预报）就像每天早上的菜市场报价单。它告诉你"今晚下 8mm"，但你站点真实下了多少？可能是 3mm（菜贩多报了），也可能是 50mm（暴雨）。

  • GRAPES 3km（中国气象局）：本地分辨率最高，山地刻画好，但有夜雨偏低、白天湿偏的系统倾向
  • ECMWF 0.125°（欧洲）：全球公认强模式，但分辨率粗，极端降水几乎全漏（TS@100mm ≈ 0）
  • 站点观测：100 个 A0001~A0100 自动站，6 小时和 12 小时累积降水

比赛要求选手在 NWP 报价的基础上做订正，输出 100 站 × 2 时段的 12 小时累积降水量。订正得好，YPSI 就高。

5YPSI 评分公式拆解

YPSI = 8 个分项加权（非微量场景）。其中 inv_wMAE 占 40%，四个 TS 合计 45%——任何只优化 MAE 的方案都会在 TS 上崩。

YPSI = 0.08·TS0.1 + 0.10·TS25 + 0.12·TS50 + 0.15·TS100
   + 0.40·inv_wMAE + 0.08·LogBias + 0.02·invF + 0.05·Δextreme

8 个分项的精确权重

分项权重公式 / 含义
TS@0.1mm8%有雨阈值命中（hits / (hits + misses + false_alarms)）
TS@25mm10%大雨阈值命中（25mm 缓冲）
TS@50mm12%暴雨阈值命中
TS@100mm15%大暴雨阈值命中（GRAPES 0.14 / ECMWF ≈ 0）
inv_wMAE40%exp(-wMAE_pro / 25)，加权 MAE 反向得分
LogBias8%1 / (1 + |log₁₀(Σp / Σo)|)，对称惩罚总量偏差
invF2%无雨站点空报惩罚（F̄ = 平均虚报值）
Δextreme5%Strend + Sheavy，极端趋势 + 强降水误差奖励

等效映射（误差软化）

评分前先对"有雨站点的高估"做一次软化：

p* = o + min(0.5·(p-o), 5) if o > 0 且 p > o
p* = p otherwise

含义：站点真实下了雨，模型多报的部分最多减分 5mm。这降低了"宁多报、不漏报"的代价。

TS 阈值化命中分类

TSτ = Hτ / (Hτ + Mτ + Fτ + ε)
H = 命中 (p≥τ ∧ o≥τ) ｜ M = 漏报 (p<τ ∧ o≥τ) ｜ F = 空报 (p≥τ ∧ o<τ)

可微版用 sigmoid 替代硬阈值：σ((p−(τ−15))/5)，并在 25/50/100mm 阈值附近给 2mm 缓冲。

语义直觉：规则鼓励"极端不要漏报"——100mm 高估额外因子仅 ×1.05，而 100mm 以下高估是 ×1.8；极端高估再 ×0.85。但反过来，到处报大雨会被 LogBias、invF、Δ 三重惩罚。这种张力决定了 V2b 必须保留 GRAPES 的极端原始信号。

8QM 分位数映射可视化

QM 的本质：把 NWP 预报的 CDF 拉伸/压缩到匹配实况的 CDF。简单但有效——绕开了 GBDT 的均值回归陷阱。

选择站点 + 月份查看映射曲线（mock data）

读图：蓝色曲线是某站某月的 QM 映射函数。曲线高于对角线 = NWP 偏低需上抬；低于对角线 = NWP 偏高需压低。红色区是极端旁路：≥50mm 直接保留 NWP 原值，避免 QM 因尾部样本稀缺把暴雨预报"杀死"。
展开：QM 实现关键代码（empirical 100 分位） 
q_pred = np.quantile(grapes_pred_train, np.linspace(0, 1, 101))
q_obs  = np.quantile(obs_train,         np.linspace(0, 1, 101))
# 推理时
def qm_apply(p, q_pred, q_obs):
    if p >= 50.0: return p          # 极端旁路
    if p < 0.5:  return p           # 微量旁路
    return np.interp(p, q_pred, q_obs)

202 个 (target_half, sid) 键，每键 (101,) × 2 数组 = 1.6 KB；总 pkl 335 KB。

失败根因（每方案点击展开）

V3 LightGBM (MSE 损失) — 0.4726

设计初衷：用站点 + NWP + 历史滑窗特征做 GBDT，目标函数 MSE。

致命问题：降水分布严重右偏（pre08_20 均值 1.91mm，但右尾到 100mm+），MSE 在右偏分布上严重均值回归——把暴雨预报压扁。最优 best_iter 接近 0，几乎等价于"不学"。

V3.5 LightGBM (YPSI 梯度) — 0.4499

设计初衷：把 YPSILoss 反传到 GBDT 自定义 obj，希望直接优化目标。

致命问题：YPSI 梯度极度稀疏（TS 部分只在阈值附近有信号），早停又依赖 wMAE → 方向错位。最终训练 loss 下降但验证 YPSI 反而劣于纯 scale baseline。

V3 CatBoost / XGBoost DART — 0.4466 / 0.4753

设计初衷：试不同 GBDT 框架；DART dropout 抑制 overfit。

共性失败根因：小样本 + 长尾的双重夹击让 boosting 优势消失。DART best_iter=0（1 棵树）被自动早停选中——"不学"反而比"乱学"好，但仍劣于 V2b 的纯统计映射。

V2c DEM-residual — 离线 +0.005 / 线上 -0.017

设计初衷：在 V2b 残差上加 6 维 DEM 特征（高程/坡度/坡向 sin·cos/TPI×2/粗糙度），用简单 MLP 学残差。

致命问题：站点孤立特征无空间上下文（一个 6 维向量代表整座山）；全 regime 混训，让中雨日被过度削减；离线提升完全是 overfit 验证集。2026-04-22 实测 -0.017。

Patch-CNN (B 方案) — 离线 +0.018 / 无法上线

设计初衷：取 GRAPES 5×5 patch + 站点 6 维 DEM，用 CNN 提空间上下文。

致命问题：2026 年没有 GRAPES 实时格点（组委会只发 12h 累积站点 NWP）→ 训练时有 patch 推理时没有，线上不可部署。即使有 patch，大雨段（>25mm）仍劣化 -0.04~-0.06。

V4 day-level routing — oracle +0.0133 / 实跑 -0.003

设计初衷：每天用 ML 模型选当日是用 ensemble 还是 single；oracle（事后最优选择）给 +0.0133。

致命问题：样本量不足（288 天），feature 噪声 >> 信号，ML routing 学不到 oracle 边界；rule-based 也只在 +0.001 附近。等正赛积累 50+ 轮再重试。

5 条系统性教训

  1. 在 GRAPES baseline 上做订正，不要从零训。raw YPSI 0.4465 已经是强基线，重学只会更差。
  2. YPSI 梯度优化远比 wMAE 困难。TS 项的硬阈值让梯度稀疏，可微版 sigmoid 也只是缓解。
  3. 极端段（≥50mm）样本稀缺阻断 ML 学习。一年 5-9 月汛期，全 100 站 ≥50mm 全历史只有 3246 次。
  4. 离线-线上差距大的方案要谨慎。V2c 离线 +0.005 但线上 -0.017，根因是验证集有 overfit。
  5. 站点孤立特征必须配空间上下文。DEM 6 维代表"一座山"，但同一山脊不同坡向降水差异大。

12FAQ + 难点小结

常见问题

为什么不用深度学习？
试过 6 个方向（V3 LightGBM / V3.5 LightGBM YPSI 梯度 / CatBoost / DART / V2c MLP / Patch-CNN）全输给 V2b。详见 §9。核心原因：YPSI 梯度稀疏 + 极端样本稀缺 + 离线-线上差距大。
QM 怎么处理新站（没训练数据的站）？
用 (target_half, '*') 全局兜底键 — 把所有站合并训练一份"通用 LUT"，新站用这份。生产 pkl 里 100 站 + 1 全局 = 101 个键 × 2 窗口 = 202。
为什么 ensemble 用 mean 而不是 median 或 trimmed_mean？
10 天 backtest 实测：mean 优于 max/median/trimmed_mean。原因可能是雅安 6 源 NWP 的偏差是系统性的而非随机的，median 截尾会丢极端信号。
GRAPES 6 月缺 76.7% 怎么办？
训练时按日期 mask 剔除（QM checkpoint 训练数据 107,942 行已扣除 6 月）；推理时 daily_submit 走 fallback：缺 GRAPES → ECMWF MICAPS → 6 源 ensemble。
能不能合并 GRAPES + ECMWF？
已合并：predict.py 里 ECMWF 用于微量降水补全——raw < 0.5 mm 且 ECMWF ≥ 1.0 mm → 抬到 ≥ 0.5 mm。这是 V2b 的"极端保护"另一面。
为什么 pre20_08 × 1.2 但 pre08_20 × 1.0？
EDA 显示 GRAPES 夜雨偏低（× 1.2 拉升）、白天湿偏（早期 × 0.5 但已纠正为 × 1.0；现在白天偏差由 QM 内部消化）。常量 1.2 来自 2024 年训练集对齐拟合。
YPSI 微量降水场景（max(o) < 1mm）权重怎么变？
10 个分项重新分配：inv_wMAE 30% + Micro_Hit 15% + Micro_Prec 10% + TS₀.₁ 20% 主导。这就是为什么"全无雨"日 niji123 排名波动大——micro_hit/precision 用 sigmoid σ(10(p−0.1))，对噪声极敏感。
每天都要提交吗？
不是。正式赛 15-20 轮按通知触发：前瞻式（提前 3 天）/ 回溯式（重大过程后）/ 突击式（<6h）。但本项目 cron 每天 14:00 自动跑，确保通知日不漏。最低出勤率 70%。

4 大难点小结

难点 1：极端样本稀缺

10 年汛期全 100 站 ≥100mm 仅 110 次。任何 ML 模型都无法用这么少样本学到稳定 mapping，必须保留 NWP 极端信号。

难点 2：YPSI 梯度稀疏

8 项加权 + 4 个阈值 TS，梯度大部分时间为 0；可微版 sigmoid 缓解但仍劣于纯统计映射。

难点 3：山地夜雨主导

85% 极端事件来自夜间 + 高海拔 + 南北跨度 194km。同一系统在不同站点持续时间差 ±2h。

难点 4：离线-线上偏差

V2c +0.005 离线却 -0.017 线上；CNN +0.018 离线但无法上线。验证集 overfit 是常态。
