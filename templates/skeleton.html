1故事:为什么会有这道题?

先不看公式,先看一个真实场景。

训练一个大模型(比如 GPT 那种),需要 几百到几千张算力卡(GPU/NPU)同时干活。这些卡 不是独立的——每过一小段计算,它们就要把"自己算出的中间结果"发给其他卡,合并、再算下一步。

这就引出问题:这么多卡同时往网络里发数据,怎么不堵?

类比想象一栋办公楼里 800 个员工同时去寄快递。如果楼下只有一个快递柜,大家要排队几小时;但如果有 32 个柜子,而且能聪明地把每个人导向不同的柜子,就几乎不用等。这道题就是给数据流找柜子。

这道题让你写一个算法:给定一张训练任务"数据流图",决定每条流走哪条上行链路(哪个端口),让全网最拥堵的那条链路尽量轻,同时尽量避免不同时间段、不同任务之间互相打架。

2网络长什么样?(Leaf-Spine 拓扑)

数据中心常用的两层网,理解它就理解了 80% 的题。

整个集群分两层交换机:

  • Leaf 交换机(下层,绿色):每个 Leaf 直连一群算力卡(p × r 张)
  • Spine 交换机(上层,紫色):连接所有 Leaf,负责跨 Leaf 转发
  • 每个 Leaf 有 p 个上行端口,分别接到 p 个 Spine

下面是 Example 1 的真实网络(l=2, p=2, r=2,共 8 张卡):

Spine 交换机 Leaf 交换机 算力卡 上行链路(端口)

关键洞察

  • 同 Leaf 内两张卡通信(比如卡 0 → 卡 1):走 Leaf 内部,不占用上行端口,在题里记为 port = -1
  • 跨 Leaf通信(比如卡 2 → 卡 6):必须走 Leaf → Spine → Leaf,必须选一个上行端口 port ∈ [0, p-1]
  • "端口分配"就是替每条跨 Leaf 流挑端口

规模上限(竞赛规则)

符号含义上限
nJob(作业)数40
lLeaf 数100
p每 Leaf 上行端口数 = Spine 数32
r带宽比 = 单端口冲突容量4
m_j每 Job 的 Phase 数31
f每 Phase 的流数12800

最坏情况共 40 × 31 × 12800 ≈ 1500 万 条流,5 秒内给每条流定 port。

4"流"是什么?怎么读输入?

一条流 = 卡 A 给卡 B 发一份数据。读法很反直觉,小心踩坑。

题目用 "src dst src dst …" 成对解析。看 Example 1 Job1 的这一行:

2 3 3 6 6 7 7 2

它表示 4 条流:

  • 流 1: 2 → 3(同 Leaf 0,内部流)
  • 流 2: 3 → 6(Leaf 0 → Leaf 1,跨 Leaf)
  • 流 3: 6 → 7(同 Leaf 1,内部流)
  • 流 4: 7 → 2(Leaf 1 → Leaf 0,跨 Leaf)
易错很多人(包括我)第一次会以为前一半是 src 列、后一半是 dst 列。不是。是 src1 dst1 src2 dst2 … 严格成对。
易错同一 Phase 里,同一张卡可以同时当一次 src 和一次 dst。例如上面卡 3:在流 1 里当 dst,在流 2 里当 src,合法。但不能当两次 src 或两次 dst。

5Job 和 Phase:层层嵌套

想象成俄罗斯套娃:训练集群 ⊃ Job ⊃ Phase ⊃ Flow。

Job(作业)
最多 n=40 个

一个独立的训练任务,有自己的卡、自己的通信模式。多个 Job 同时跑在集群上,互相 独立 但 共享网络。

Phase(阶段)
每 Job 最多 m=31 个

同一 Job 内部按时间分阶段。同一 Phase 内的流"同时发生",不同 Phase 串行。

Flow(流)
每 Phase 最多 f=12800 条

最小单元:卡 A → 卡 B 一份数据。你的算法要给每条 跨 Leaf 流分配一个端口。

在线决策
读完一 Job → 立刻输出

这是个在线问题:决定 Job1 的端口时,看不到 Job2、Job3 长什么样。

一个 Job 的输入示例(Example 2 Job1)

1 4                  ← 1 个 Phase, 4 条流
0 4 4 0 1 5 5 1      ← Phase 1 的 4 条流: (0,4)(4,0)(1,5)(5,1)
在线协议·全题最大难点题目协议是交互式流: 
读 Job 1 输入 → 立刻输出 Job 1 端口分配 → 读 Job 2 输入 → 立刻输出 Job 2 ...
决定 Job 1 时看不到 Job 2/3 的内容。但 Job 间冲突恰恰需要"全局协调",这就是这道题难的根源——你必须用一个能与"未来未知 Job"兼容的策略。

6所以你要做什么?

一句话:对每条跨 Leaf 流,选一个 port ∈ [0, p-1]。

对 Example 1 Job1 的 4 条流,完美的端口分配是这样:

对应的输出:

4
2 3 -1   3 6 1   6 7 -1   7 2 1
↑       ↑        ↑        ↑
内部 -1  跨 Leaf 选 port 1  内部 -1   跨 Leaf 选 port 1

这个分配让 2 条跨 Leaf 流都走 port 1,看起来"挤一起",但因为 r=2(端口能扛 2 条流),没问题。实际上这是题面给的满分解。

承上启下从第 7 章开始,我们只关心一件事:每个 (Leaf, port, 方向) 上同时挂了多少条流。超过 r 的部分 = 冲突。下面把"冲突"按发生时机拆成三类。

7什么算"堵车"?三类冲突详解

堵车的代价是要扣分的。三种冲突权重不一样,Phase 内最严重。

Phase 内冲突 ×12
最严重
Phase 间冲突 ×5
中等
Job 间冲突 ×3
最轻
前置约定下面所有"端口"都按 (Leaf, port, 方向 in/out) 三元组分别计——一条 src→dst 流同时占用 (srcLeaf, port, out) 和 (dstLeaf, port, in) 两条记录。下面 r=2 默认。

7.1 Phase 内冲突 C_inphsc(权重 ×12)

定义:同一个 Job 的同一个 Phase 内,某个 (Leaf, port, 方向) 上挂的流数 v > r 时,超出量 v − r 计入。

for each (Job j, Phase k, Leaf i, Port p, dir ∈ {out, in}):
    v = 该端口该方向的流数
    if v > r:
        C_inphsc += v - r

例子

某 Phase 中 (Leaf0, port 0, out) 上有 4 条流向外发的流 → 4 − 2 = 2 计入 C_inphsc。

类比 — 电梯瞬时超载

电梯额定 r=2 人,你硬塞 4 人 → 超载 2 人,瞬时冲突。第 3、4 个人就是 C_inphsc。

7.2 Phase 间冲突 C_btphsc(权重 ×5)

定义:同一个 Job 内,同一张源卡(同一个 src)在相邻 Phase 选了不同的 port。每发生一次切换,C_btphsc += 1。

易错这不是"占用累加超 r"。它惩罚的是同源流跨阶段换路——硬件上每次换路都要做端口切换/缓冲腾挪,有真实开销。 
for each Job j:
    for each Phase k > 0:
        for each src 卡 s:
            prev_port = s 在 Phase k-1 的输出 port  (跳过 -1)
            this_port = s 在 Phase k   的输出 port  (跳过 -1)
            if prev_port ≠ this_port:
                C_btphsc += 1

例子

Job 1 的卡 5 在 Phase 1 把流 5→11 走 port 0,在 Phase 2 把流 5→9 走 port 1 → 卡 5 这一次切换计 C_btphsc += 1。

类比 — 快递员临时换路线

同一个快递员上午走 A 路、下午改走 B 路 — 每次换路要重新规划 + 装车,产生切换成本。

7.3 Job 间冲突 C_bttskc(权重 ×3)

定义:某条流的端口被它所在 Phase 的负载 + 其他 Job 在该端口的 max-Phase 之和累加超出 r 时,这条流计 1。

两个关键口径
  1. 每个 Job 取它在该端口的 max Phase(因为不同 Job 之间不知道彼此 Phase 对齐,只能取最坏情形)
  2. 是命中流数,不是超出量——只要某条流"踩中"超载端口,这条流就 +1,不管端口超了多少
// 先对每个 Job 算它在每个 (Leaf, port, dir) 上的 max-Phase 流数 jmax[j]
// 全局累加: multi[(L,p,dir)] = Σ_j jmax[j][(L,p,dir)]

for each Job j:
    for each Phase k:
        for each flow (s, d) in Phase k:
            ext_out = multi[(srcLeaf, port, out)] - jmax[j][(srcLeaf, port, out)]
            ext_in  = multi[(dstLeaf, port, in)]  - jmax[j][(dstLeaf, port, in)]
            if phase_load_out + ext_out > r  OR  phase_load_in + ext_in > r:
                C_bttskc += 1     ← 命中流 +1

例子

设 (Leaf 2, port 1, out) 上累加:

  • Job 1: max-Phase = 0 条
  • Job 2: max-Phase = 2 条
  • Job 3: max-Phase = 2 条
  • multi = 0 + 2 + 2 = 4 > r=2 → 端口超载

对 Job 2 的 2 条流:本 Phase 负载 2 + 其他 Job(=Job 3) max=2 → 4 > r=2,这 2 条全部命中,C_bttskc += 2;Job 3 同理 +2。Job 1 这条端口上没流,不命中。

8端口负载比 Maxsingle / Maxmulti

这是评分里的大头——80 分都来自这两个比值。

对每个端口(每个 Leaf 的 p 个上行口),统计上面挂了多少条流。除以 r,就是它的"负载比"。比值越接近 1.0 越好。

Maxsingle_r:单 Job 内最大端口比

Maxsingle_r = max(1.0, Maxsingle_c / r)
Maxsingle_c = max over (Job j, Phase k, Leaf i, Port p, dir ∈ {out, in})
              [ 该 (Leaf, port, dir) 在该 Phase 的流数 ]

每个 Job 内部独立看,所有 (Leaf, port, 方向) 三元组里最挤的那个。反映 单个任务 占用网络的均匀度。

Maxmulti_r:全局最大端口比

Maxmulti_r = max(1.0, Maxmulti_c / r)
Maxmulti_c = max over (Leaf i, Port p, dir ∈ {out, in})
             [ Σ_j 各 Job 的 max-Phase 流数 ]

所有 Job 叠加之后,所有 (Leaf, port, 方向) 三元组里最挤的那个。反映 整个集群 的拥堵峰值。

关键max(1, ·) 截断意味着流数 ≤ r 时该项就拿满 40 分,不会因为"端口很空"额外加分;比值 > 1 时按反比例曲线扣分(见下一章)。

9评分公式:每一项含义拆解

三类冲突扣分 + 两个端口比加分,合起来满分 100。

Score = 20 − (12·C_inphsc + 5·C_btphsc + 3·C_bttskc) / 总流数 + 40 / Maxsingle_r + 40 / Maxmulti_r

逐项解释

项 取值范围 含义
20 常数项 = 20 无条件基础分,送的
− 12·C_inphsc / N ≤ 0 Phase 内冲突惩罚,N 是总流数;每条 Phase 内冲突扣 12/N
− 5·C_btphsc / N ≤ 0 Phase 间冲突惩罚;每条 Phase 间冲突扣 5/N
− 3·C_bttskc / N ≤ 0 Job 间冲突惩罚;每条 Job 间冲突扣 3/N
+ 40 / Maxsingle_r (0, 40] 单 Job 端口均衡奖励,Maxsingle=1 时给满 40
+ 40 / Maxmulti_r (0, 40] 全局端口均衡奖励,Maxmulti=1 时给满 40
坑题面 PDF OCR 显示像"−5·B − 3·C",但从 Example 2 Output1 反算 20 − 8·3/12 + 40/1 + 40/2 = 78 可知三项都是惩罚(分子前都是减号),且 20 不参与除法。

满分长什么样?

  • 三类冲突全为 0,惩罚项 = 0
  • Maxsingle = Maxmulti = 1.0 → 加分 40 + 40
  • 20 + 0 + 40 + 40 = 100 ✓

各项最坏情况

  • 三类冲突最多让前面变成 20 − (12·N)/N = 20 − 12 = 8(理论极限,几乎不会发生)
  • 端口比通常上限 ~r·总流数/总端口数;比如 Maxmulti = 4,加分项变成 40/4 = 10

10反比例曲线:1.0 → 1.5 比 1.5 → 2.0 更要命

为什么算法工程师对端口比格外敏感?因为 40/x 是反比例函数,边际收益不均匀。

评分里 40 / Maxsingle 和 40 / Maxmulti 都是反比例函数。看下面这个曲线就懂了:

拖一拖看曲线点:
x = 2.00, 得分 = 40/x = 20.00, 扣分 = 20.00

关键观察

  • 端口比从 1.0 升到 1.5:扣 13.3 分(40 → 26.7)
  • 端口比从 1.5 升到 2.0:只扣 6.7 分(26.7 → 20)
  • 端口比从 2.0 升到 3.0:再扣 6.7 分(20 → 13.3)

言外之意:

  1. 把已经很好的端口比从 1.0 提升到 1.0 边际收益最大——但也最难,因为已经接近完美
  2. 同样道理,从 2.0 救回 1.5 比从 3.0 救回 2.5 收益更大
  3. 算法应该优先打掉那些 1.0–1.5 区间的端口(虽然听起来反直觉,因为它们看起来已经"还行")
实战经验仓库的算法用 40/peak² 作为边际导数估计,优先消除那些"端口比刚冒头超过 1"的端口,而不是去救已经爆掉的端口——后者怎么救都救不到 1.0,但前者动一下就能直接拉满。

11算分演示:78 分是怎么来的?

把所有概念拼一起,完整走一遍 Example 2 的 78 分解。

题目回顾

Example 2 的网络:n=3 个 Job, l=3 个 Leaf, p=2 端口, r=2,卡 0–11(每 Leaf 4 张)。

三个 Job 各 1 Phase 4 条全跨 Leaf 流:

  • Job 1: (0,4)(4,0)(1,5)(5,1) — Leaf0 ↔ Leaf1
  • Job 2: (2,10)(10,2)(3,11)(11,3) — Leaf0 ↔ Leaf2
  • Job 3: (6,8)(8,6)(7,9)(9,7) — Leaf1 ↔ Leaf2

总流数 N = 12。

Output 1(78 分)的端口分配

这是 OJ 题面给的参考解之一。每个 Job 把 4 条流全部挤到同一个端口(单看自己 OK,但叠加就翻车):

JobLeaf 对分配
Job 1L0 ↔ L14 条流全走 port 0
Job 2L0 ↔ L24 条流全走 port 1
Job 3L1 ↔ L24 条流全走 port 1
关键口径端口负载按 (Leaf, port, 方向) 三元组分别计——一条 src→dst 流同时占用 (srcLeaf, port, out) 和 (dstLeaf, port, in) 两条记录。后面所有 peak / Maxsingle / Maxmulti / C 都基于这个三元组。

Step 1 — 数 Phase 内冲突 C_inphsc

Job 1 的 4 条流全走 port 0,按方向拆开:

  • (L0, p0, out) = 2 条 (0→4, 1→5 从 L0 出去)
  • (L0, p0, in) = 2 条 (4→0, 5→1 进入 L0)
  • (L1, p0, out) = 2;(L1, p0, in) = 2

每个方向最多 2 条 = r,刚好不超 → C_inphsc = 0。Job 2/3 同理。

Step 2 — 数 Phase 间冲突 C_btphsc

每 Job 只有 1 个 Phase,不存在跨 Phase → C_btphsc = 0

Step 3 — 数 Job 间冲突 C_bttskc

对每个 (Leaf, port, 方向),把所有 Job 的 max-Phase 流数累加:

(Leaf, port, 方向)Job 1Job 2Job 3合计超 r=2?
(L0, p0, out/in)2002✓ 不超
(L1, p0, out/in)2002✓ 不超
(L0, p1, out/in)0202✓ 不超
(L1, p1, out/in)0022✓ 不超
(L2, p1, out)0224✗ 超 2
(L2, p1, in)0224✗ 超 2
官方口径C_bttskc 不是"超出量",而是 "命中超载端口的流数"。
Job 2 的 4 条流都途径 (L2, p1, out/in),被 Job 3 累加导致总数 4 > r=2 → Job 2 的 4 条全部计入 C_bttskc。Job 3 同理 4 条全部计入。

C_bttskc = Job 2 命中 4 + Job 3 命中 4 = 8(Job 1 没和任何 Job 撞端口,4 条流都不计入)

Step 4 — 求 Maxsingle

每个 Job 单看,所有 (Leaf, port, 方向) 上流数最多是 2 → Maxsingle_c = 2,除以 r=2 → Maxsingle_r = 1.0

Step 5 — 求 Maxmulti

叠加后 (L2, p1, out) 和 (L2, p1, in) 各有 4 条流,Maxmulti_c = 4 → Maxmulti_r = 4 / 2 = 2.0

Step 6 — 组装总分

基础分+ 20
Phase 内惩罚: 12 × 0 / 12− 0
Phase 间惩罚: 5 × 0 / 12− 0
Job 间惩罚: 3 × 8 / 12− 2.0
单 Job 端口比: 40 / 1.0+ 40.0
全局端口比: 40 / 2.0+ 20.0
合计78.0 分

如果把 Maxmulti 从 2.0 救回 1.0,这一项就从 +20 变成 +40,加上消除 Job 间冲突,刚好就是满分 100。这就是 Output 3 的故事。

12同一题,不同算法的差距(Example 2)

点切换看三种端口分配如何拿到 78、83.7、100 三种分数。

三种解法的差异完全不在"会不会做",而在"能不能做出全局协调":

所有流都挤同一端口(蠢办法)

每个 Job 把它的 4 条跨 Leaf 流"对称地"分到 port 0、port 1。各自看是平衡的(Maxsingle=1.0),但叠加后某些 (Leaf, port) 上挂了 4 条流(2+2),Maxmulti = 4/2 = 2.0。

Score = 20 − 3·8/12 + 40/1.0 + 40/2.0 = 20 − 2 + 40 + 20 = 78.0

直觉错误:每个 Job 独立看都"完美",但合起来烂。这就是为什么不能单 Job 贪心。

稍微错开(中等)

三个 Job 之间有部分错开,Maxmulti 降到 1.5,但局部冲突还多了一些。

Score = 20 − 3·12/12 + 40/1.0 + 40/1.5 = 20 − 3 + 40 + 26.67 = 83.67

趋势对,但没做到极致。

完美错开(满分)

三个 Job 用同一种"约定":源在 Leaf 偶数序号的卡 → port 0,奇数序号 → port 1。这样每个 Leaf 的 port 0、port 1 在 3 个 Job 叠加之后,负载都正好 = r。

Job流→port
Job 1 (0↔4, 1↔5)0↔4 用 port 0,1↔5 用 port 1
Job 2 (2↔10, 3↔11)2↔10 用 port 0,3↔11 用 port 1
Job 3 (6↔8, 7↔9)6↔8 用 port 0,7↔9 用 port 1
每个 Leaf 的 port 0:2 条流(满 r=2)→ 比值 1.0 每个 Leaf 的 port 1:2 条流(满 r=2)→ 比值 1.0 Score = 20 − 0 + 40/1.0 + 40/1.0 = 100.0

本质是"全局协调"——每个 Job 单独决策时无所谓,但合起来能不能不打架,取决于算法是否提前约定了某种全局模式。这就是这道题难的地方:你看不到后面的 Job,但要做出能与未来兼容的决策。

13FAQ:你可能想问的 N 个问题

各种容易误解、容易踩坑的细节。

Q1: 为什么同 Leaf 内通信不占端口?

Leaf 交换机本身有"环回交换能力"——卡 0 给卡 1 发数据,Leaf 内部直接转发,根本不用上 Spine。所以这种流标记 port = -1,不计入任何端口冲突。

Q2: 为什么端口冲突要除以 r?r 是什么?

r 是上下行带宽比。每个 Leaf 的下行总带宽是 p × r 个单位(p 张卡每张 r),上行总带宽是 p 个端口每个 r 单位。即一个端口能容纳 r 条并发流而不冲突。
题面 r ≤ 4,意味着一个端口最多扛 4 条流。

Q3: 一条流被分到 port 后,意思是什么?

port 编号代表选择第 k 个 Spine。计分时同时占用源 Leaf 的出方向 (srcLeaf, port, out) 和目的 Leaf 的入方向 (dstLeaf, port, in) 两条记录。换 port 等于走另一台 Spine 转发。

Q4: 为什么 Phase 内冲突权重最高(×12)?

因为 Phase 内冲突是"实时"的——同一时刻多个流抢同一个端口,会真的造成丢包/重传。其他冲突或多或少是"时间错位的潜在影响",代价较小。

Q5: 在线问题?那我能不能存所有 Job 一次决定?

不能。题目协议是读完 Job1 输入 → 立即输出 Job1 的端口分配 → 再读 Job2。如果你不输出,程序就 hang 在那里。所以必须做"在线决策":此刻不知未来,但要做出对未来友好的选择。

Q6: 5 秒是 wall clock 还是 CPU?平台是什么硬件?

5 秒是题面规定的算法时间上限(wall);OJ 系统不杀进程的容忍上限是 100 秒(包含编译)。机器规格 1 Core CPU + 2GB 内存——单核,所以多线程没用、只能拼算法。

Q7: Maxsingle 和 Maxmulti 都是 max,那如果某个端口爆了,Maxmulti=4,直接给我扣 30 分?

对,而且 Maxmulti 通常比 Maxsingle 大,因为它是跨 Job 累加。算法的目标之一就是压低这个全局峰值,而不是只优化单 Job 视角。

Q8: 总流数 N 是不是只算跨 Leaf 流?

不,N 是所有流(包括 port=-1 的内部流)。这是分母,所以 N 越大,每条冲突摊薄后的扣分越小。这意味着 N 大的题"容错率"更高。

Q9: 题目允许重复流吗?(同一 Job 出现 (A,B) 两次)

同一 Job 内允许跨 Phase 有相同 (src, dst);输出时按 (src, dst) 唯一化(去重)。同一 Phase 内不会有完全重复的 (src, dst) 出现两次。

Q10: 算法跑超 5 秒会怎样?

题面 5 秒是 deadline;100 秒是平台杀进程门槛。仓库当前算法实测 max case 单核约 4.6–4.8s,留一点 buffer 让算法在没结束时也能立刻输出当前最佳解。超时 = 0 分。

Q11: 比值小于 1.0 怎么算?

比值会先 max(1, peak/r) 截断到 1.0,所以 40/x 上限是 40。这意味着"端口很空闲"不给额外奖励,只奖励"刚好填满不超载"。

Q12: 不同 Leaf 的同号端口算冲突吗?(Leaf0 的 port0 和 Leaf1 的 port0)

不算。冲突按 (Leaf, port, 方向) 三元组分别累计——同 Leaf 同 port 的 out 和 in 还要再分。所以 Leaf0/p0/out 和 Leaf1/p0/out 互不影响,Leaf0/p0/out 和 Leaf0/p0/in 也分别计。

14整道题的难点在哪?

现在回头看,你应该明白这是个「在线决策 + 全局协调」问题。

四个核心难点

  1. 在线性:决定 Job1 的端口时,看不到 Job2、Job3 长什么样。但 Job 间冲突恰恰需要全局协调。
  2. 规模:最大 ~1500 万条流要在 5 秒内决策,几乎没时间"试错"。
  3. 多目标:同时要降三类冲突 + 同时降两个端口比,这些目标之间会打架。比如想让 Maxmulti 平,可能要牺牲一些 Phase 内对齐。
  4. 评分公式高度非线性:40/peak 是反比例函数,从 1.0 → 1.5 是 −13 分,从 1.5 → 2.0 才 −7 分,边际收益不均匀,不能用线性 cost 函数代替。
